## 系统概述

### 1.1 项目背景

“共享雨伞”旨在解决高校师生突遇降雨时的出行痛点，通过类似"共享充电宝"的模式，实现校园雨具的 **24小时无人值守** 借还服务。

### 1.2 系统目标

- **用户端**：提供便捷的自助借还服务，支持多终端并发登录
- **管理端**：提供完整的运营管理功能，包括雨具调度、站点监控和数据统计
- **技术目标**：采用分层架构，确保代码可维护性和可扩展性

### 1.3 技术栈

- **开发语言**：C++17
- **UI框架**：Qt 6.9.3 (Widgets)
- **数据库**：MySQL 8.0
- **构建工具**：CMake 3.16+
- **编译器**：MinGW 11.2+ 或 MSVC 2019+

### 1.4 文档编写说明（范围界定）
**本文档主要侧重于系统核心架构设计与关键业务逻辑的实现分析。**
出于篇幅与阅读效率的考虑，文档中的后端与前端设计章节选取了**具代表性的核心模块**（如：多线程连接池等）进行详细阐述，对于基础的 CRUD 操作及相似性代码不做全量展示。

## 系统架构设计

### 2.1 整体架构

系统采用经典的**分层架构**（Layered Architecture），将业务逻辑、数据访问和表现层清晰分离：

**架构层次结构：**

```
表现层 (UI Layer)
├── 客户端UI (Client UI)
└── 管理端UI (Admin UI)
         ↓
业务逻辑层 (Service Layer)
├── AuthService      (认证服务)
├── BorrowService    (借还服务)
└── StationService   (站点服务)
         ↓
数据访问层 (DAO Layer)
├── UserDao      (用户数据访问)
├── GearDao      (雨具数据访问)
├── StationDao   (站点数据访问)
└── RecordDao    (记录数据访问)
         ↓
数据模型层 (Model Layer)
├── User         (用户模型)
├── RainGear     (雨具模型)
└── Stationlocal (站点模型)
         ↓
数据库 (MySQL)
```

### 2.2 架构优势

1. **职责分离**：每一层只负责自己的职责，降低耦合度
2. **易于测试**：可以独立测试每一层的功能
3. **易于维护**：修改某一层不影响其他层
4. **可扩展性**：可以轻松添加新的Service或DAO

## 数据库设计

### 3.1 数据库概述

数据库名称：`rainhub_db`  
字符集：`utf8mb4`  
存储引擎：`InnoDB`

### 3.2 数据表设计

#### 3.2.1 用户表 (users)

| 字段名 | 类型 | 说明 |
|--------|------|------|
| user_id | VARCHAR(20) | 主键，学工号 |
| password | VARCHAR(64) | 密码 |
| real_name | VARCHAR(20) | 真实姓名 |
| role | INT | 角色：0=学生, 1=教职工, 9=管理员 |
| credit | DECIMAL(10,2) | 账户余额，默认0.00 |
| is_active | TINYINT(1) | 是否激活：0=未激活, 1=已激活 |

**设计说明**：
- `is_active` 字段用于首次登录流程：未激活用户需要设置密码
- `credit` 用于存储押金和租金，支持充值功能

#### 3.2.2 站点表 (station)

| 字段名 | 类型 | 说明 |
|--------|------|------|
| station_id | INT AUTO_INCREMENT | 主键，对应 Station 枚举 |
| name | VARCHAR(50) | 站点名称 |
| pos_x | FLOAT | 地图X坐标 |
| pos_y | FLOAT | 地图Y坐标 |
| status | INT | 在线状态：1=在线, 0=离线 |
| unavailable_slots | VARCHAR(50) | 故障槽位，逗号分隔，如 "1,5,8" |

**设计说明**：
- `pos_x` 和 `pos_y` 用于地图可视化
- `unavailable_slots` 存储故障槽位，避免用户选择

#### 3.2.3 雨具表 (raingear)

| 字段名 | 类型 | 说明 |
|--------|------|------|
| gear_id | VARCHAR(20) | 主键，RFID标识 |
| type_id | INT | 类型：1=普通塑料伞, 2=高质量抗风伞, 3=专用遮阳伞, 4=雨衣 |
| station_id | INT | 所属站点ID（外键） |
| slot_id | INT | 槽位编号：1-12 |
| status | INT | 状态：0=Unknown, 1=Available, 2=Borrowed, 3=Broken |

**槽位分配规则**：
- 槽位 1-4：普通塑料伞
- 槽位 5-8：高质量抗风伞
- 槽位 9-10：专用遮阳伞
- 槽位 11-12：雨衣

#### 3.2.4 借还记录表 (record)

| 字段名 | 类型 | 说明 |
|--------|------|------|
| record_id | BIGINT AUTO_INCREMENT | 主键 |
| user_id | VARCHAR(20) | 用户ID（外键） |
| gear_id | VARCHAR(20) | 雨具ID（外键） |
| borrow_time | DATETIME | 借出时间 |
| return_time | DATETIME | 归还时间（NULL表示未归还） |
| cost | DECIMAL(10,2) | 使用费用，默认0.00 |

**设计说明**：
- `return_time` 为 NULL 表示未归还，用于查询用户当前借出的雨具
- `cost` 在归还时计算并更新

### 3.3 索引设计

- `users.role`：按角色查询用户
- `raingear.station_id`：按站点查询雨具
- `raingear.status`：按状态查询雨具
- `record.user_id`：查询用户的借还记录
- `record.gear_id`：查询雨具的借还历史

### 3.4 外键约束

- `raingear.station_id` → `station.station_id` (ON DELETE SET NULL)
- `record.user_id` → `users.user_id` (ON DELETE RESTRICT)
- `record.gear_id` → `raingear.gear_id` (ON DELETE RESTRICT)

### 4.3 业务逻辑层 (Service Layer)

#### 4.3.1 AuthService - 认证服务

**职责**：处理用户登录、密码验证、账户激活

```cpp
class AuthService {
public:
    enum class LoginStatus {
        SuccessNormal,      // 非首次登录
        SuccessFirstTime,   // 首次登录需设置密码
        UserNotFound,       // 用户不存在
        NameMismatch,       // 姓名不匹配
        DatabaseError,      // 数据库错误
        AdminNotAllowed     // 管理员不能在客户端登录
    };
    
    LoginStatus checkLogin(const QString& id, const QString& name);
    bool verifyPassword(const QString& id, const QString& password);
    bool activateUser(const QString& id, const QString& name, const QString& password);
};
```

**业务流程**：
1. 用户输入学号和姓名
2. `checkLogin()` 验证学号和姓名是否匹配
3. 检查 `is_active` 字段：
   - 未激活 → 跳转首次登录页面
   - 已激活 → 跳转密码输入页面
4. `verifyPassword()` 验证密码
5. 登录成功，返回用户对象

#### 4.3.2 BorrowService - 借还服务

**职责**：处理借伞、还伞的核心业务逻辑

```cpp
class BorrowService {
public:
    struct ServiceResult {
        bool success;
        QString message;
        double cost = 0.0;
    };
    
    ServiceResult borrowGear(const QString& userId, const QString& gearId);
    ServiceResult returnGear(const QString& userId, const QString& gearId, Station stationId, int slotId);
private:
    double calculateCost(const QDateTime& borrowTime, const QDateTime& returnTime, GearType type);
};
```

**借伞流程**：
1. 检查用户是否存在且已激活
2. 检查用户是否有未归还的订单
3. 检查雨具是否存在且状态为 Available
4. 检查用户余额是否足够支付押金
5. **开启事务**：
   - 扣除用户押金
   - 更新雨具状态为 Borrowed
   - 创建借出记录
6. 提交事务或回滚

**还伞流程**：
1. 验证槽位是否合法（根据雨具类型）
2. 检查槽位是否被占用
3. 查找用户的未归还订单
4. 验证归还的雨具ID是否匹配
5. 计算使用费用（按小时计费）
6. **开启事务**：
   - 更新记录（归还时间、费用）
   - 更新雨具状态为 Available，设置新位置
   - 退还押金（扣除费用后）
7. 提交事务或回滚

**计费规则**：
- 普通塑料伞：1元/小时
- 高质量抗风伞：2元/小时
- 专用遮阳伞：1.5元/小时
- 雨衣：2元/小时
- 费用最多等于押金，不会倒扣用户余额

### 4.4 工具层 (Utils Layer)

#### 4.4.1 ConnectionPool - 数据库连接池

**设计**：采用**线程局部存储（Thread-Local Storage）**机制

```cpp
class ConnectionPool {
public:
    static QSqlDatabase getThreadLocalConnection();
    static void removeThreadConnection();
};
```

**实现原理**：
1. 每个线程通过 `QThread::currentThreadId()` 获取唯一标识
2. 为每个线程创建独立的数据库连接
3. 连接名格式：`Conn_{threadId}`
4. 线程结束时调用 `removeThreadConnection()` 清理

**优势**：
- **线程安全**：每个线程拥有独立连接，避免并发冲突
- **性能优化**：连接复用，减少创建/销毁开销
- **简单高效**：无需复杂的连接池管理逻辑

- **状态颜色**：
  - 🟢 绿色：可借（有雨具且状态为Available）
  - ⚪ 灰色：空槽可还（无雨具）
  - 🔴 红色：故障（槽位在unavailable_slots中）

**交互流程**：

**借伞模式**：
1. 用户点击绿色槽位
2. 系统获取该槽位的雨具ID
3. 调用 `BorrowService::borrowGear()`
4. 显示结果提示
5. 刷新槽位状态

**还伞模式**：
1. 用户点击灰色槽位
2. 系统获取用户当前借出的雨具
3. 验证槽位是否合法（根据雨具类型）
4. 调用 `BorrowService::returnGear()`
5. 显示费用和退款信息
6. 刷新槽位状态

**自动刷新**：
- 使用 `QTimer` 每3秒刷新一次槽位状态
- 确保多用户并发时数据实时性

## 技术实现细节

### 7.1 事务管理

所有涉及多表更新的操作都使用数据库事务：

```cpp
if (!db.transaction()) {
    return {false, "事务开启失败"};
}

bool success = true;
if (!operation1()) success = false;
if (!operation2()) success = false;
if (!operation3()) success = false;

if (success) {
    db.commit();
    return {true, "操作成功"};
} else {
    db.rollback();
    return {false, "操作失败"};
}
```

**优势**：确保数据一致性，要么全部成功，要么全部回滚。

### 7.2 工厂模式 - 雨具创建

```cpp
class RainGearFactory {
public:
    static std::unique_ptr<RainGear> create(
        const QString& id, 
        GearType type, 
        GearStatus status, 
        Station station, 
        int slot
    ) {
        switch (type) {
            case GearType::StandardPlastic:
                return std::make_unique<StandardPlasticUmbrella>(id, status, station, slot);
            case GearType::PremiumWindproof:
                return std::make_unique<PremiumWindproofUmbrella>(id, status, station, slot);
            // ... 其他类型
        }
    }
};
```

**优势**：封装对象创建逻辑，便于扩展新的雨具类型。

### 7.3 异常处理

- **数据库连接失败**：返回明确的错误信息
- **SQL执行失败**：记录日志并回滚事务
- **业务逻辑验证失败**：返回友好的提示信息

### 7.4 日志记录

使用 Qt 的 `qDebug()`, `qInfo()`, `qWarning()`, `qCritical()` 分级记录：
- 信息日志：连接成功、操作成功
- 警告日志：数据异常但可恢复
- 错误日志：操作失败、事务回滚

## 性能优化策略

### 8.1 数据库优化

1. **连接池**：线程局部存储，避免频繁创建连接
2. **索引优化**：在常用查询字段上建立索引
3. **查询优化**：只查询必要字段，避免 `SELECT *`

### 8.2 前端优化

1. **页面懒加载**：页面在需要时才创建
2. **数据缓存**：静态数据（站点坐标）预加载
3. **定时刷新**：合理设置刷新间隔，避免过度查询
4. **UI响应**：耗时操作在后台线程执行

### 8.3 混合数据加载策略

**地图页面**：
- 静态数据（站点坐标、名称）→ 从JSON文件加载（毫秒级）
- 动态数据（库存数量）→ 从数据库查询（实时）

**优势**：
- 降低数据库I/O压力
- 提升页面加载速度
- 保证数据实时性

## 总结

本系统采用**分层架构**设计，前后端职责清晰，代码结构良好。后端通过DAO模式实现数据访问的独立性，Service层封装业务逻辑，前端通过信号槽机制实现页面间通信。系统支持多终端并发，通过线程局部存储的数据库连接池确保线程安全。整体设计兼顾了可维护性、可扩展性和性能优化。
